## 摘要
语言模型代理通过多轮工具使用在解决现实任务中越来越有效。然而，训练可靠的工具使用代理在实践中仍然具有挑战性。虽然强化学习提供了一种基于策略的范式，使代理能够从自身环境交互中改进，但其有效性在很大程度上取决于训练任务的分布。当任务在训练前固定时，任务分布可能与策略不断发展的能力越来越不匹配，导致许多轨迹浪费在无信息量的任务上。我们提出SENTINEL，一种失败驱动的强化学习框架，将求解器的轨迹失败转化为针对性的训练任务。SENTINEL遵循控制器-提议器-求解器循环：控制器分析失败轨迹并总结重复出现的错误模式，提议器生成强调这些弱点的可执行任务，求解器在针对性任务上训练。在Tau2-Bench Retail上使用Qwen3-4B-Thinking-2507，SENTINEL将Pass^1从66.4提升到74.9，并且在Pass^k指标上优于一般合成任务上的强化学习。这些结果表明，模型失败为改进工具使用语言模型代理提供了有效且可扩展的针对性训练信号来源。

## Abstract
Language model agents are increasingly effective in solving realistic tasks through multi-turn tool use. However, training reliable tool-using agents remains challenging in practice. While reinforcement learning provides an on-policy paradigm for improving agents from their own environment interactions, its effectiveness depends heavily on the training task distribution. When tasks are fixed before training, the task distribution can become increasingly mismatched with the policy's evolving capabilities, causing many rollouts to be spent on uninformative tasks. We propose SENTINEL, a failure-driven reinforcement learning framework that turns the Solver's rollout failures into targeted training tasks. SENTINEL follows a Controller--Proposer--Solver loop: the Controller analyzes failed trajectories and summarizes recurring error patterns, the Proposer generates executable tasks that stress these weaknesses, and the Solver is trained on the targeted tasks. On Tau2-Bench Retail with Qwen3-4B-Thinking-2507, SENTINEL improves Pass\^{}1 from 66.4 to 74.9 and outperforms RL on general synthetic tasks across Pass\^{}k metrics. These results demonstrate that model failures provide an effective and scalable source of targeted training signal for improving tool-using language model agents.

- **核心问题**：训练可靠的工具使用语言模型代理（LLM agents）时，强化学习（RL）的有效性严重依赖于训练任务的分布。当训练任务在训练开始前被固定下来后，随着策略模型能力的不断演化，固定的任务分布会变得越来越不匹配，导致大量训练交互（rollouts）浪费在无信息量的任务上。
- **研究动机**：现有RL方法大多依赖静态任务集，但“太简单、太难、重复或与模型实际弱点不匹配”的任务会浪费资源，甚至鼓励模型利用奖励或环境中的伪特征，而不是学习鲁棒的工具使用行为。
- **整体含义**：论文提出应让训练任务分布随策略模型的学习状态自适应调整，利用模型自身的失败轨迹作为诊断和训练信号的桥梁，实现“从失败中学习”的闭环。

## 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：构建一个 **Controller–Proposer–Solver** 迭代循环。  
  - **Controller**：分析Solver的失败轨迹，总结重复出现的错误模式（如错误物品选择、缺失必要动作、策略违规等），并生成用于引导任务生成的指令。
  - **Proposer**：根据Controller指令，首先生成一个可执行的参考工具调用轨迹（基于环境），再将轨迹重写为自然语言用户任务，使新任务专门针对当前策略的弱点。
  - **Solver**：在包含原始任务和新生成任务的缓冲区上进行RL训练（使用GRPO算法），更新策略。新策略的失败轨迹再反馈给Controller，形成闭环。
- **关键技术细节**：
  - 轨迹采样与去重：Controller定期触发，对失败轨迹去重，并同时采样成功轨迹以防止生成已掌握任务的重复。
  - 失败模式分析：LLM分析器根据失败的对话历史、期望动作、实际工具调用、奖励分解等，输出分类错误模式（如missing_write_action, wrong_item_selection等）。
  - 指令生成：Controller维护跨轮次的训练历史，区分持续失败与已解决的问题，并生成自然语言指令传递给Proposer。
  - 任务生成：Proposer采用“轨迹先构建后重写”的两步法，先通过工具环境生成可执行轨迹，再根据Controller指令（例如强调相似产品变体的区分）将轨迹重写为用户请求，确保任务可执行且目标明确。
  - 奖励设计：主奖励为任务成功信号（数据库状态检查+通信检查），附加动作级认证检查、重复工具调用惩罚、token长度惩罚、格式惩罚等，防止奖励篡改。

## 6. 论文的主要结论与发现

1. SENTINEL在Pass^1上将66.4提升至74.9，Pass^2和Pass^3也显著高于通用RL baseline，说明失败驱动的任务生成显著提升了RL训练效率。

4. 模型失败是有效且可扩展的针对性训练信号来源。

## 7. 优点：方法或实验设计上的亮点

- **闭环自适应机制**：将失败诊断与任务生成、RL优化无缝连接，使得训练数据分布随模型能力动态演化，避免了固定任务集带来的资源浪费。
- **轨迹先决后重写**：Proposer先生成可执行轨迹再写用户请求，保证了任务的可执行性和参考答案的真实性，减少了LLM直接生成时的不一致风险。
- **奖励塑形丰富**：除任务成功奖励外，引入动作级验证、重复工具调用惩罚等，有助于防止RL训练中的退化行为（如死循环、格式错误）。
- **在更强模型上仍有效**：SFT后应用SENTINEL依然能提升，而通用RL反而下降，这强有力地证明了针对性训练的优势。

## 8. 不足与局限：实验覆盖、偏差风险、应用限制

- **实验覆盖**：仅在一个受控零售环境（Tau2-Bench Retail）上验证，缺乏其他工具使用领域（如软件工程、医疗、金融）的泛化性验证。不同领域的工具种类、策略复杂度、错误模式可能差异很大。
- **偏差风险**：Controller的错误分析依赖于LLM，可能引入语言模型自身的偏差或漏判。错误模式类别（表2）由人工预定义，可能遗漏未见错误类型。
- **应用限制**：
  - 任务覆盖率受限于当前Solver暴露的失败模式，对于罕见或采样不足的场景可能无法覆盖。
  - 框架复杂度较高（需要维护Controller、Proposer、Solver三个角色），增加了系统实现和调参的难度。
  - 奖励设计中对“重复工具调用”等行为的惩罚阈值需要手动设定，不同环境可能需要重新调整。
- **计算开销**：每轮需要运行Solver rollout、Controller分析、Proposer生成新任务，多次迭代可能带来较大计算成本（论文未详细报告总资源消耗）。
- **未提供统计显著性检验**：缺少多次重复实验的标准差，难以判断改进是否稳定。
