## 摘要
LLM智能体在真实应用（如网页搜索和深度研究）中日益面临长周期任务，累积的上下文会导致长上下文退化和推理失败。先前的工作通过智能体侧的上下文控制或固定策略（如摘要）来缓解这一问题，这需要训练智能体自身进行适应——使得该方法对于闭源智能体不切实际，且忽略了不同智能体可能需要不同策略。我们提出自适应上下文管理（AdaCoM），该方法通过灵活的修改操作和端到端强化学习，训练一个外部LLM来管理冻结智能体的上下文。在网页搜索和深度研究基准测试中，针对不同智能体，AdaCoM通过保留任务约束和进展、同时剪枝过时内容，显著提升了性能。学习到的策略揭示了保真度-可靠性权衡：基础ReAct性能较高的智能体受益于高保真度的上下文保留，而性能较低的智能体则需要更激进的压缩以保持在可靠推理范围内。迁移实验表明，AdaCoM在能力相似（由基础ReAct性能衡量）的智能体间泛化效果最佳，这为智能体系统提供了一条通往可复用上下文管理器的实用路径。

## Abstract
LLM agents increasingly face long-horizon tasks such as web search and deep research in real-world applications, where accumulated context can cause long-context degradation and reasoning failures. Prior work mitigates this through context management with agent-side context control or fixed strategies such as summarization, which require training the agent itself for adaptation - making it impractical for closed-source agents and ignoring that different agents may require different strategies. We introduce Adaptive Context Management (AdaCoM), which trains an external LLM to manage the context of a frozen agent through flexible modification actions and end-to-end reinforcement learning. Across diverse agents on web search and deep research benchmarks, AdaCoM substantially improves performance by preserving task constraints and progress while pruning stale content. The learned strategies reveal a Fidelity-Reliability Trade-off: agents with higher vanilla ReAct performance benefit from higher-fidelity context preservation, whereas lower-performing agents require more aggressive compression to stay within a reliable reasoning regime. Transfer experiments show that AdaCoM generalizes most effectively across agents with similar capability (measured by vanilla ReAct performance), suggesting a practical path toward reusable context managers for agent systems.

- **现有工作局限**：以往方法要求智能体自身进行上下文管理（如提示其生成摘要或调用剪枝工具），这需要训练或微调智能体，不适用于闭源模型（如 GPT-4o、Claude 等）；同时，固定策略（如统一摘要）忽视不同智能体在架构、训练数据、推理风格上的巨大差异，无法实现“因agent而异”的最优管理。
- **本文目标**：提出一种**无需训练智能体本身**的外部上下文管理框架，通过学习发现每个冻结智能体最兼容的上下文管理策略。

## 2. 方法论

### 核心思想
- **架构解耦**：上下文管理由一个**外部管理器**（通常是一个较小的 LLM）负责，被管理的智能体完全保持冻结（frozen）。
- **操作级灵活性**：管理器不限于预定义操作（如摘要），而是可以自由地对上下文中的任意消息进行**重写、删除、合并或保留**。动作表示为 JSON 格式的修改操作列表。

### 关键技术细节
1. **工作流程**：
   - 在每步智能体动作前，管理器先根据当前上下文生成修改操作 \( m_t \)。
   - 将修改应用到上下文后，得到管理后的上下文 \( \tilde{c}_t \)。
   - 智能体基于 \( \tilde{c}_t \) 生成下一步动作 \( a_{t+1} \)。
2. **动作空间**：
   - \( m_t = [\delta^{(1)}_t, \delta^{(2)}_t, \dots] \)，每个 \( \delta \) 包含：目标消息 ID、角色（system/user/assistant）、理由（justification）、新内容（new_content，空则删除）。
3. **训练流程**：
   - **SFT 热身**：先用 GPT-5 或 Claude Opus 4.6 生成编辑轨迹，让管理器（Qwen3-4B-Instruct）学习输出格式。
   - **强化学习（RL）**：基于 **GRPO**（Group Relative Policy Optimization）和 PPO 风格的截断目标优化。组大小 \( G=8 \)。
4. **奖励设计**：
   - **结果奖励**：最终答案是否正确（LLM Judge 打分）。
   - **过程奖励**：规则驱动，包括令牌惩罚（上下文超限）、冗余动作惩罚（重复调用工具）、格式惩罚（JSON 解析失败）等。在 BrowseComp-Plus 上还加入“发现金/关键文档”的正奖励。
5. **两级优势估计**：
   - 任务级优势：对组内结果奖励归一化。
   - 步骤级优势：对组内过程奖励归一化。
   - 两者加权（α=0.1）后再进行组内归一化，提高训练稳定性。

## 6. 主要结论与发现

1. **性能提升显著**：AdaCoM 在 BrowseComp-Plus上平均相对提升 39%（mean@3），所有智能体均受益；Kimi-K2-Instruct 提升高达 95%。
2. **保真度-可靠性权衡（Fidelity–Reliability Trade-off）**：
   - 基础 ReAct 性能强的智能体（如 GLM、Qwen）倾向于保留更多原始上下文（高保真），管理器干预更晚更轻。
   - 基础性能弱的智能体（如 DeepSeek、Kimi）需要更激进的压缩（及时蒸馏），以保持推理可靠性。
   - 固定摘要（SumCoM）更接近激进压缩策略，因此对弱智能体有效，但可能损害强智能体。
3. **跨智能体迁移规律**：
   - 管理器在**能力相近**的智能体间迁移效果最好（能力由 vanilla ReAct 性能衡量）。
   - 也存在例外：如 DeepSeek 从 Kimi 管理器中获益有限，因为两者虽然能力相似但偏好的记忆风格不同（前者简练、后者详细）。
4. **失败模式缓解**：
   - 减少了约束遗忘（通过维护任务状态记录）、过早放弃（通过保留搜索进展）、冗余探索（通过剪枝重复工具调用结果）。

## 7. 优点

- **架构解耦**：无需训练或修改智能体本身，适用于任何闭源或黑箱模型，实践意义强。
- **操作高度灵活**：非固定策略，可学习重写、删除、合并等任意修改，自适应不同智能体的需求。
- **训练稳定**：两级优势估计 + 过程奖励设计，缓解长程稀疏奖励问题，提升训练效率。
- **可迁移性**：训练好的管理器可在能力相似的智能体间复用，节省部署成本。
- **分析深入**：不仅报告指标提升，还揭示策略差异中的“保真度-可靠性”权衡，并给出实际部署指导。

## 8. 不足与局限

- **评估范围有限**：仅测试了网页搜索和知识检索场景，未涉及代码智能体、具身智能体、长文本写作等，这些场景中上下文结构可能完全不同。
- **深度研究实验规模小**：MCP-Bench-Wiki 仅用 2 个智能体，结论的泛化性受限。
- **管理器容量限制**：管理器为 4B 模型，对于非常强的智能体（如 GPT-5 或 Claude Opus），可能无法无损保留或压缩细粒度信息，导致收益不充分。
- **推理开销**：每步多一次管理器推理，增加延迟和 token 消耗。虽然强智能体场景下管理器介入频率低，但未量化端到端开销。
- **KV 缓存失效**：上下文修改会破坏 KV 缓存重用，降低推理效率。该局限与大多数上下文管理方法共享，本文未专门优化。
- **资源信息缺失**：未报告 GPU 型号、数量、训练总时长，不利于复现和成本估算。
